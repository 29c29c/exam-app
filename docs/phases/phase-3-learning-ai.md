# 阶段 3：学习流程完善与 AI 能力增强

## 1. 阶段目标

本阶段重点不再是“把题库导进来”，而是“让题库真正好用起来”。

本阶段完成后，应达到以下结果：

- 做题流程更完整
- 用户能按目标筛选题目
- 收藏与学习进度形成闭环
- AI 解析体验更稳定

## 2. 本阶段边界

本阶段要做：

- 做题页功能增强
- 题目筛选和搜索
- 随机模式和答题进度
- 收藏与学习状态扩展
- AI 服务封装
- 批量 AI 解析优化

本阶段不做：

- 回收站
- 完整自动化测试体系
- 大规模运维改造

## 3. 做题页增强方案

### 3.1 题目获取能力

当前题目接口应升级为支持查询参数：

- `page`
- `pageSize`
- `keyword`
- `type`
- `bookmarked`
- `hasAnalysis`
- `random`

示例：

- `GET /api/banks/:id/questions?keyword=唯物&type=single_choice`

### 3.2 筛选能力

建议做题页支持：

- 全部题目
- 仅收藏
- 仅未解析
- 仅有解析
- 按题型筛选

### 3.3 随机模式

增加：

- 顺序模式
- 随机模式

实现方法：

- 前端拿到题目后打散
- 或后端按随机顺序返回

建议先在前端实现，成本更低。

### 3.4 答题卡增强

答题卡建议增加状态：

- 当前题
- 已收藏
- 已查看答案
- 已掌握

如果引入“已掌握”功能，建议新增表：

- `question_progress`

字段建议：

- `user_id`
- `question_id`
- `mastery_status`
- `last_viewed_at`
- `view_count`

## 4. 搜索与学习状态

### 4.1 搜索

建议支持：

- 题干关键词搜索
- 答案搜索
- 来源搜索

SQLite 第一阶段可先使用 `LIKE`，后续如数据量大再考虑全文索引。

### 4.2 学习状态

建议增加：

- 标记已掌握
- 标记稍后再看
- 最近查看

这样可在后续支持：

- 复习模式
- 仅看未掌握
- 学习统计

## 5. AI 服务重构

### 5.1 目标

当前 AI 逻辑可用，但应从路由中抽离，形成稳定服务层。

建议新增：

- `server/services/ai-service.js`

### 5.2 能力拆分

建议拆成：

- provider 适配
- prompt 构造
- 错误处理
- 结果保存

### 5.3 provider 适配层

建议统一对外暴露：

```js
analyzeQuestion({
  provider,
  apiKey,
  question,
  answer
})
```

内部再按 provider 分派。

### 5.4 批量 AI 解析优化

当前做法是前端逐题串行调用。

可优化为两步：

#### 方案 A：保留同步串行，但封装更稳

适合当前体量。

改进点：

- 增加失败重试
- 增加中断恢复
- 增加成功/失败数量统计

#### 方案 B：改为后端任务模式

适合后续阶段。

实现：

- `POST /api/ai/analyze-batch`
- 返回 `taskId`
- 前端轮询任务状态

建议本阶段优先实现方案 A，预留方案 B 的接口设计。

## 6. AI 配置管理建议

### 6.1 当前问题

当前 API Key 存前端 `localStorage`，仅适合单机原型。

### 6.2 阶段建议

本阶段先不强行改为服务端托管，但要做两件事：

- 增加明确提示，说明 Key 仅保存在本地浏览器
- 把 provider 和 key 的读写封装成独立配置模块

可新增：

- `src/utils/ai-config.js`

## 7. 学习体验优化

建议增加的轻量功能：

- 重新生成 AI 解析
- 复制 AI 解析
- 折叠/展开解析
- 快速跳题
- 回到上次进度

## 8. 验收标准

本阶段完成后，应满足：

- 做题页支持搜索、筛选、随机
- 用户可以标记收藏和学习状态
- AI 解析逻辑有独立服务层
- 批量 AI 解析比当前更稳定

## 9. 建议拆分任务

1. 扩展题目查询接口
2. 增加做题页筛选条件
3. 增加随机模式
4. 增加学习状态记录
5. 抽离 AI 服务层
6. 优化批量 AI 解析流程

## 10. 阶段完成标志

当下面条件满足时，可认为阶段 3 完成：

- 用户已能围绕题目完成较完整的学习闭环
- AI 解析不再只是“能调接口”，而是具备基本稳定性和可维护性
